# Sand Together — Stage 3 实现 Brief:压缩归档(无限堆积)

> 给"专门做 Stage 3 的新 session"的起步说明:目标、现状、设计要点、增量步骤、要守的约束都在这。
> 配套阅读:`game-design.md`(沉积/成就)、`architecture.md`(分层/数据流)、`backend.md`(协议/参数)、`CLAUDE.md`(硬约束)、记忆 `dev-workflow`(部署)。

## 目标

让瓶子能**无限往上堆**,而不靠把网格高度 `H` 无限加大。做法:**活动区固定大小 + 深层老沙压缩成薄层归档**,向下滚动可看到(以后可展开)历史。这就是用户那句"一张满了存下来、继续堆"的正解——存的方式是"压缩成薄层",不是真换一张网格。

## 现状(Stage 2,起点)

- 服务端权威:每房间一张固定 `W=80 × H=300` 网格,跑 falling-sand,广播增量 patch,存盘 `server/data/<room>.json`。
- 客户端纯渲染:收 snapshot/patch,画 `viewRows=250` 行(`S=2` → 500px),相机把堆顶锚在 0.618。
- **痛点**:网格堆满(沙到第 0 行)就停,新沙放不进来。`H` 有限,加大只是治标 + 占内存/带宽。

## 核心思路:两段式存储

1. **活动网格(active grid)**:固定高度(沿用现在的 `H`,或单独设 `ACTIVE_H`),只保留**堆顶一段**的全分辨率沙,用来跑物理 + 全分辨率渲染。
2. **归档带(bands)**:活动网格底部"落定不动"的老沙,压缩成**很薄的层**(一条 band 概括很多行),按时间顺序往下堆,单独存储。

当活动堆高超阈值时:取**最底部**已落定的若干行 → 压成一条 band → 追加进归档 → 活动网格整体下移、腾出顶部空行继续接新沙。视觉上就是"无限往上长"。

## 设计要点(新 session 里定稿)

- **band 存什么**(隐私红线照旧:只存计数/像素统计,**绝无键位内容/文本**):建议每条 band 记 `rows`(压缩了多少行)+ 每列的颜色/密度摘要(例如把那段每列降采样成 1 行的颜色,或每列各颜色计数)。够还原出一条有颜色质感的细带即可。
- **触发 & 压缩**:活动堆顶逼近活动网格顶部 N 行内 → 把最底部 K 行压成一条 band(N、K 实现时调)。
- **协议扩展**(`backend.md`):snapshot 增加 `bands`(归档列表);新建 band 时广播一条 `band` 增量消息;客户端活动网格照旧全分辨率渲染,归档带画在活动区**下方**,每条 1~数 px 高的彩色细条。
- **相机**:可滚动范围 = 活动区 + 归档总高;向下拖拽进归档区看历史。可选二期:进入某条 band 时**展开**还原细节(`game-design.md` 的"向下滑展开");先做"压缩细条堆叠"即可。
- **持久化**:存盘 = 活动网格 + bands 列表。向后兼容:老存档只有网格、无 bands → bands 视为空。
- **带宽**:bands 一次性下发 + 偶发新增,稳态几乎不占带宽;活动区仍只发变化单元。

## 要守的约束(`CLAUDE.md`)

- 服务端权威物理,客户端纯渲染——**压缩也在服务端做**。
- 隐私红线:client / server / 存档**只有计数与像素统计,绝无键位内容**。
- 纯 HTML/CSS/vanilla JS,无框架、无构建步骤。

## 建议增量步骤(每步可独立测/部署)

1. **服务端**:加 bands 数据结构 + "满了压底部成 band + 活动网格下移"逻辑 + 存盘 + snapshot/广播带上 bands。先不动客户端,用 `server/smoke.mjs` 验证 bands 正确生成/持久化。
2. **客户端**:处理 snapshot/`band` 消息 → 活动区下方渲染细条;相机范围延伸到归档。浏览器两窗口一致性验证。
3. **(可选二期)** 向下滚动展开某条 band 的交互。

## 测试

- 扩展 `server/smoke.mjs`:狂灌输入直到触发压缩,断言 bands 产生、活动网格下移、新沙能继续进。
- 浏览器两窗口看归档一致。

## 部署提醒

这是**后台 + 共用协议**改动 → 上线要 `git pull` **+ `systemctl restart titb`**(VPS `/opt/titb`,分支 `main`)。前后端 `W/H/常量`、协议字段必须同步改。
